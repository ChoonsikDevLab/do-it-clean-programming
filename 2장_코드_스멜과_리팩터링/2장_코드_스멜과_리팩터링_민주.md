# 2장. 코드 스멜과 리팩터링

## 2-7. 기능 편애 (Feature Envy) 제거하기

### 개념
어떤 함수가 자신이 속한 클래스보다 **다른 클래스의 데이터나 함수에 더 관심을 갖는 상태**를 의미함.

### 리팩터링 방법
1. **참조 부분 단순화:** 다른 클래스의 개별 필드를 직접 참조하는 대신, 해당 클래스에 합산/계산 함수를 추가하여 참조 횟수 자체를 줄임. (사이드 이펙트 위험이 적어 선호됨)
2. **함수·필드 이동:** 참조 대상 함수나 필드를 소유 클래스로 아예 이동시킴. (타 모듈 참조 시 사이드 이펙트 주의)

### 프로젝트 적용 사례

**Before: 서비스가 엔티티의 데이터에 너무 큰 관심을 가짐**
`ReservationServiceImpl.java`의 `updateReservation` 메서드에서 서비스 로직이 `Reservation` 엔티티를 일일이 검사하며 필드를 수정하고 있었음. (결합도 ↑)

```java
boolean updated = false;

if (request.getPatientName() != null) {
    reservation.updatePatientName(request.getPatientName());
    updated = true;
}
if (request.getPatientAge() != null) {
    reservation.updatePatientAge(request.getPatientAge());
    updated = true;
}
// ... 이와 같은 if문이 7개 연속으로 등장 ...

```

**After: 객체 스스로 업데이트하도록 책임 위임**
데이터를 가지고 있는 객체(`Reservation`)가 스스로 자신을 업데이트하도록 변경함. 서비스는 비즈니스 흐름만 제어하게 되어 응집도는 높아지고 결합도는 낮아짐.

```java
// 1. Reservation.java (엔티티 내부로 로직 이동)
public boolean updateFromRequest(ReservationUpdateRequest request) {
    boolean updated = false;
    if (request.getPatientName() != null) {
        this.patientName = request.getPatientName();
        updated = true;
    }
    // ... 나머지 필드 업데이트 로직 ...
    return updated;
}

// 2. ReservationServiceImpl.java (서비스 호출부는 깔끔해짐!)
boolean updated = reservation.updateFromRequest(request);

if (updated) {
    reservation.setUpdatedAt(LocalDateTime.now());
    // 시간 검증 로직 등...
}

```

## 2-8. 과다한 매개변수 줄이기

### 리팩터링 방법

1. **중복된 매개변수 제거하기:** 다른 값으로 계산 가능한 매개변수는 내부에서 직접 계산하도록 변경함.
2. **플래그 매개변수 제거하기:** `boolean` 매개변수를 넘기는 대신, 별도 함수로 분리하여 의도를 명확히 함.
3. **매개변수 객체화하기:** 관련된 매개변수들을 하나의 클래스(DTO 등)로 묶어서 전달함.

### 프로젝트 적용 사례

**Before: 15개의 매개변수 전달**
`ReservationServiceImpl.java`에서 객체를 생성할 때 무려 15개의 매개변수를 넘기고 있었음. 순서가 헷갈려 버그가 발생할 위험이 높음.

```java
Reservation reservation = new Reservation(
        guardianId,
        request.getGuardianName(),
        request.getCaregiverId(),
        request.getCaregiverName(),
        request.getPatientName(),
        request.getPatientAge(),
        // ... (총 15개)
);

```

**After: DTO 객체 자체를 전달 (Factory 메서드 활용)**
이미 관련된 데이터가 뭉쳐진 `ReservationCreateRequest` 객체를 활용하여 생성자의 책임을 깔끔하게 만듦.

```java
// Reservation 엔티티 내부에 팩토리 메서드 구현
public static Reservation createFrom(UUID guardianId, ReservationCreateRequest request) {
    return new Reservation(
            guardianId,
            request.getGuardianName(),
            request.getCaregiverId(),
            // ... 내부에서 request.get() 처리
    );
}

// ReservationServiceImpl.java 호출부 (15개 -> 2개로 압축!)
Reservation reservation = Reservation.createFrom(guardianId, request);

```

## 2-9. 매직 넘버 제거하기

### 개념

소스 코드 내에 의미 없이 하드코딩된 숫자를 의미함. 코드를 읽는 사람이 숫자의 의미를 파악하기 어려우므로, 의미 있는 이름의 상수(Constant)로 변경해야 함.

### 프로젝트 적용 사례

**Before: 의미를 알 수 없는 숫자와 문자열**
기본 날짜 범위를 설정하기 위해 `1`이라는 숫자가 반복적으로 사용되고, 결제 서비스에서는 하드코딩된 문자열과 숫자 `8`이 등장함.

```java
// ReservationServiceImpl.java
if (startDate == null) {
    startDate = LocalDateTime.now().minusMonths(1); // 1 = 1개월? 1년? 의미가 숨겨져 있음
}

// PaymentServiceImpl.java
Payment payment = new Payment(..., "TOSS_PAYMENTS"); // 하드코딩된 결제 수단 문자열

actualResult.setApprovalNumber(request.getApprovalNumber() != null ? 
    request.getApprovalNumber() : "TEST-" + UUID.randomUUID().toString().substring(0, 8)); // 8은 왜 8인가?

```

**After: 의미 있는 상수로 치환**
클래스 상단에 의미 있는 이름의 상수(`static final`)로 선언하여 가독성을 높이고, 기획 변경 시 수정이 용이해짐.

```java
// 클래스 상단 상수 선언
private static final int DEFAULT_SEARCH_MONTHS_RANGE = 1;
private static final String DEFAULT_PG_PROVIDER = "TOSS_PAYMENTS";
private static final int TEST_ID_LENGTH = 8;
private static final String TEST_PREFIX = "TEST-";

// 사용부
if (startDate == null) {
    startDate = LocalDateTime.now().minusMonths(DEFAULT_SEARCH_MONTHS_RANGE);
}

Payment payment = new Payment(..., DEFAULT_PG_PROVIDER);

String fallbackApprovalNumber = TEST_PREFIX + UUID.randomUUID().toString().substring(0, TEST_ID_LENGTH);
actualResult.setApprovalNumber(
    request.getApprovalNumber() != null ? request.getApprovalNumber() : fallbackApprovalNumber
);

```
